# 部署验证指南

本文档用于指导当前项目进行一次可操作的部署验证，目标是让项目不仅能在本机运行，还能在同一局域网中的另一台设备上访问和演示。

## 1. 验证目标

- 本机可以正常启动数据库、中间件、后端服务、前端服务
- 另一台设备可以通过浏览器访问前端页面
- 前端页面可以正常调用后端接口
- 另一台设备可以完成登录和主要业务演示

当前阶段的验证定位：

- 这是“局域网部署验证”
- 不是正式生产部署
- 当前更接近“联调环境 / 演示环境 / 试运行环境”

## 2. 当前推荐验证方式

结合当前项目结构，推荐使用以下方式进行验证：

- `Docker Compose` 启动 PostgreSQL、Redis、MinIO
- `NestJS` 后端运行在本机 `3001` 端口
- `Vite` 前端运行在本机 `5173` 端口，并通过开发代理转发 `/api`
- 另一台设备通过本机局域网 IP 访问前端页面

推荐这个方式的原因：

- 当前前端接口地址是相对路径 `/api`
- 当前代理逻辑已经写在 `frontend/vite.config.ts`
- 不需要额外先搭建 Nginx，就能完成一轮真实可访问验证

## 3. 验证前提

开始前确认以下条件：

- 两台设备处于同一局域网
- 本机已安装 Node.js、npm、Docker Desktop
- Docker Desktop 已启动
- 项目依赖已经安装完成

如果是第一次在这台机器上执行，请先完成：

```powershell
npm.cmd install
```

## 4. 启动依赖服务

在项目根目录执行：

```powershell
docker compose -f infra\docker-compose.yml up -d
```

执行后应至少看到以下服务处于运行状态：

- `postgres`
- `redis`
- `minio`

可使用以下命令确认：

```powershell
docker compose -f infra\docker-compose.yml ps
```

## 5. 初始化数据库

如果是首次验证，或者希望刷新演示数据，请执行：

```powershell
npm.cmd run prisma:generate
npm.cmd run prisma:migrate
npm.cmd run prisma:seed
```

说明：

- `prisma:migrate` 会把当前 Prisma 迁移应用到本地数据库
- `prisma:seed` 会写入中文演示账号、学生数据、政策数据、通知数据

## 6. 启动后端

在项目根目录执行：

```powershell
npm.cmd run dev:backend
```

默认监听端口：

- `3001`

后端健康检查地址：

```text
http://127.0.0.1:3001/api/health
```

如果本机浏览器可以返回健康检查结果，说明后端启动正常。

## 7. 启动前端并暴露到局域网

前端必须使用 `--host 0.0.0.0` 启动，否则默认只允许本机访问。

在项目根目录执行：

```powershell
npm.cmd --workspace frontend run dev -- --host 0.0.0.0 --port 5173
```

默认访问地址：

- 本机访问：`http://127.0.0.1:5173`
- 局域网访问：`http://<本机局域网IP>:5173`

## 8. 获取本机局域网 IP

在本机执行：

```powershell
ipconfig
```

找到当前正在使用网卡的 IPv4 地址，例如：

```text
192.168.1.23
```

则另一台设备应访问：

```text
http://192.168.1.23:5173
```

## 9. 另一台设备访问验证

在另一台设备浏览器中完成以下验证：

### 9.1 打开首页

访问：

```text
http://<本机局域网IP>:5173
```

验证点：

- 页面可以打开
- 页面文案为中文
- 可以看到首页、学生端、管理端、领导端入口

### 9.2 登录验证

访问登录页并使用以下账号：

- 管理员：`demo.admin / demo1234`
- 学生：`demo.student / demo1234`
- 教师：`demo.teacher / demo1234`

验证点：

- 可以成功登录
- 登录后自动进入对应页面

### 9.3 学生端验证

使用学生账号登录后，至少检查：

- 当前用户信息是否显示
- 学生画像简介是否显示
- 当前流程阶段是否显示
- 提醒事项是否显示
- 我的通知是否显示
- 政策检索是否返回结果

### 9.4 管理端验证

使用管理员账号登录后，至少检查：

- 学生导入面板是否可见
- 政策知识库录入是否可用
- 通知发布是否可用
- 操作日志是否可见

建议最少执行一轮操作闭环：

1. 新增一条政策
2. 发布一条通知
3. 切换到学生账号刷新页面
4. 确认学生端能看到新增通知

## 10. 本机防火墙放行

如果另一台设备打不开页面，优先检查是否被 Windows 防火墙拦截。

建议至少放行前端端口：

```powershell
netsh advfirewall firewall add rule name="Vite 5173" dir=in action=allow protocol=TCP localport=5173
```

如果还需要让另一台设备直接访问后端接口，可再放行：

```powershell
netsh advfirewall firewall add rule name="Nest 3001" dir=in action=allow protocol=TCP localport=3001
```

## 11. 常见问题排查

### 11.1 另一台设备无法打开前端页面

重点检查：

- 前端是否使用了 `--host 0.0.0.0`
- 访问的是否是正确的局域网 IPv4 地址
- 两台设备是否处于同一局域网
- Windows 防火墙是否已放行 `5173`
- 路由器是否开启了 AP 隔离

### 11.2 页面能打开，但接口报错

重点检查：

- 后端是否已启动
- `http://127.0.0.1:3001/api/health` 是否正常
- 前端开发服务器是否仍在运行
- `frontend/vite.config.ts` 中 `/api` 代理是否保持默认配置

### 11.3 数据为空或演示内容不完整

重点检查：

- 是否执行过 `npm.cmd run prisma:seed`
- 是否已经切换到最新代码
- 数据库是否被旧测试数据污染

必要时可重新执行：

```powershell
npm.cmd run prisma:seed
```

## 12. 推荐验证顺序

建议按以下顺序执行一次完整验证：

1. 启动 Docker 依赖服务
2. 执行 Prisma 迁移与种子
3. 启动后端
4. 启动前端并暴露局域网访问
5. 本机先自测首页与登录
6. 在另一台设备访问首页
7. 在另一台设备登录学生账号验证学生端
8. 在另一台设备登录管理员账号验证管理端
9. 执行一次“发布通知 -> 学生查看通知”的闭环操作

## 13. 当前验证结论标准

满足以下条件即可认为本轮部署验证通过：

- 前端和后端都能稳定启动
- 局域网另一台设备可以访问前端页面
- 登录功能正常
- 学生端可查看流程、画像、通知、政策检索结果
- 管理端可执行导入、政策录入、通知发布
- 管理端操作能够在学生端产生可见结果

## 14. 下一阶段建议

完成本轮局域网部署验证后，下一步可以继续推进：

- 使用 `vite build` + 静态文件服务做准生产验证
- 引入 Nginx 统一代理前后端
- 增加 HTTPS、域名和进程守护
- 补充正式部署脚本与环境变量模板
