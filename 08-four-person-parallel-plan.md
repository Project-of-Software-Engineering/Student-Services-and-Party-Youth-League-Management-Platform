# 四人并行开发分工方案

## 1. 目标

把当前还未补全的需求拆成 4 个可并行推进、互相少抢文件的工作包，做到：

- 每个人都有明确主责范围
- 后端、前端、数据、联调都能同时推进
- 共享文件尽量只由一个人统一维护
- 工作量大致均衡，避免有人等接口、有人等页面

## 2. 现状判断

结合当前仓库，主要缺口集中在这些地方：

- `backend/src/modules/approvals` 还是空壳
- `backend/src/modules/files` 还是空壳
- `frontend/src/pages/LeaderDashboardPage.vue` 还是占位页
- 学生端、管理端已有骨架，但审批、统计、导出、附件、权限细化还没闭环
- `backend/prisma/schema.prisma` 和 `backend/prisma/seed.js` 还需要继续补数据模型和演示数据

所以这次分工不建议再按“谁写前端、谁写后端”来切，而是按“业务域 + 页面落点”切，避免互相等。

## 3. 分工原则

1. 以业务域切分，不按单个文件乱切。
2. 每个人都拿到一条完整链路：后端模块 + 前端页面 + 必要 DTO/种子数据。
3. 共享文件统一由 1 号维护，其他人只提需求，不直接抢改。
4. 接口和字段先对齐，再写实现。
5. 每天固定一次接口同步，每次只合并自己主责范围内的改动。
6. 每个人工作量控制在 25% 左右，允许上下浮动 5%。

## 6. 每人任务清单

### 6.1 1号：平台底座与统一鉴权

负责把所有人共用的入口先搭稳。

任务：

1. 完成登录、`/auth/me`、token 管理、路由守卫。
2. 统一用户、角色、状态返回结构。
3. 完成操作日志查询与写入的基础能力。
4. 统一 `AppShell`、登录页、首页、会话存储。
5. 整理共享接口字段和错误返回约定。

验收点：

- 管理员、教师、学生能正确登录
- 未登录时不能直接进业务页
- 登录后能按角色进入对应页面
- `logs/recent` 能查到其他模块的写操作

接口边界：

| 接口 | 方法 | 用途 |
| --- | --- | --- |
| `/auth/login` | `POST` | 登录并返回 token 与用户信息 |
| `/auth/me` | `GET` | 获取当前用户 |
| `/users` | `GET` | 管理端查看用户与角色 |
| `/logs/recent` | `GET` | 查询最近操作日志 |

### 6.2 2号：政策知识库与党团流程

负责把 P0 的“问政策、看流程”先做成完整闭环。

任务：

1. 完成政策文档列表、筛选、问答接口。
2. 完成政策文档新增/维护的基础能力。
3. 完成党团流程阶段展示和节点提醒。
4. 完成学生端政策检索区和流程时间线。
5. 把流程状态和提醒文案做成稳定映射。

验收点：

- 输入关键词能查到政策
- 输入问题能返回可读答案
- 学生端能看到当前流程阶段
- 学生端能看到下一步提醒

接口边界：

| 接口 | 方法 | 用途 |
| --- | --- | --- |
| `/policies` | `GET` | 政策列表、关键词、分类筛选 |
| `/policies` | `POST` | 新增政策文档记录 |
| `/policies/ask` | `GET` | 政策问答 |
| `/processes/my` | `GET` | 当前学生党团流程与提醒 |

### 6.3 3号：学生资料与通知运营

负责管理端最常用的“导入、维护、发布”链路。

任务：

1. 完成学生导入、增量更新、画像维护。
2. 完成学生标签、荣誉、实践、简介的维护。
3. 完成通知发布、目标标签筛选、发布记录。
4. 完成管理端首页的学生/通知/日志主面板。
5. 把导入和发布动作写入日志。

验收点：

- 导入一批学生后能查到新数据
- 学生画像和标签能显示
- 发布通知后学生端能看到
- 管理端能看到最近日志和发布结果

接口边界：

| 接口 | 方法 | 用途 |
| --- | --- | --- |
| `/students` | `GET` | 学生列表 |
| `/students/me` | `GET` | 当前学生基础信息 |
| `/students/me/profile` | `GET` | 当前学生画像 |
| `/students/:id/profile` | `GET/PATCH` | 查看和维护学生画像 |
| `/students/import` | `POST` | 批量导入或更新学生 |
| `/notices/my` | `GET` | 学生端我的通知 |
| `/notices/published` | `GET` | 管理端发布记录 |
| `/notices/publish` | `POST` | 发布通知 |

### 6.4 4号：审批流、文件服务、领导看板

负责把目前最空的模块补成可演示闭环。

任务：

1. 完成审批单创建、流转、退回、通过、驳回。

验收点：

- 能完成审批流转
- 能上传并关联附件
- 领导端能看到汇总数据

接口边界：

## 7. 共享文件统一归口

下面这些文件建议只由 1 号统一改，其他人通过接口需求单或字段草案协作：

| 文件 | 统一负责人 | 其他人协作方式 |
| --- | --- | --- |
| `backend/prisma/schema.prisma` | 1号 | 提交自己的模型字段草案，不直接改主文件 |
| `backend/prisma/seed.js` | 1号 | 提交演示数据片段和字段说明 |
| `frontend/src/router/index.ts` | 1号 | 提交新增路由需求 |
| `frontend/src/services/http.ts` | 1号 | 提交拦截器或认证需求 |
| `frontend/src/stores/session.ts` | 1号 | 提交会话字段需求 |
| `frontend/src/components/AppShell.vue` | 1号 | 提交布局和导航需求 |
| `frontend/src/types/common.ts` | 1号 | 提交公共类型需求 |

这样做的目的很直接：把冲突最高的文件收口到一个人，避免 4 个人同时改同一个入口。

## 8. 分支和合并规则

建议按负责人创建 4 条长期功能分支：

| 负责人 | 分支名 | 合并节奏 |
| --- | --- | --- |
| 1号 | `feature/platform-auth-log` | 最先合并，作为其他分支基线 |
| 2号 | `feature/policy-process` | 完成 P0 学生端后合并 |
| 3号 | `feature/student-notice-admin` | 完成管理端导入和通知后合并 |
| 4号 | `feature/approval-file-leader` | 完成审批主链路后合并 |

合并规则：

1. 共享文件必须先找 1 号对齐，不能直接抢改。
2. 每个 PR 只包含自己主责模块，不混入 unrelated formatting。
3. 合并前至少跑一次前端 build 和后端 build。
4. 后端接口变更必须在 PR 描述里写清楚请求参数、响应字段和权限。
5. 如果需要改 Prisma schema，先把字段草案发给 1 号，由 1 号统一开迁移。

## 9. 推荐推进节奏

### 第 1 阶段：接口对齐，0.5 天

1. 1 号先把登录、角色、共享布局、基础响应定下来。
2. 2、3、4 号根据自己的主责范围列出接口清单和字段清单。
3. 由 1 号统一整理共享 schema 和 seed 的变更窗口。

### 第 2 阶段：并行开发，2 到 3 天

1. 2 号先做政策和流程的只读闭环。
2. 3 号先做学生导入、画像、通知发布。
3. 4 号先把审批流和附件能力搭起来。
4. 1 号同步补权限、路由、日志和统一入口。

### 第 3 阶段：联调合并，1 天

1. 先联调 1 号的鉴权和共享入口。
2. 再联调 2 号的学生端 P0。
3. 然后联调 3 号的管理端运营链路。
4. 最后联调 4 号的审批和领导看板。

### 第 4 阶段：演示修补，0.5 到 1 天

1. 补演示数据。
2. 补空状态、加载态、错误提示。
3. 检查移动端和桌面端页面。
4. 整理验收说明和演示路径。

## 10. 每日同步格式

每天同步建议只回答 4 个问题，控制在 10 分钟内：

1. 我昨天完成了什么。
2. 我今天要完成什么。
3. 我需要谁提供什么字段、接口或数据。
4. 我今天是否会改共享文件。

## 11. 适合的验收顺序

建议按下面顺序验收，最容易看出闭环：

1. 登录与权限
2. 政策查询与流程提醒
3. 学生导入与通知发布
4. 审批流、附件和领导看板

## 12. 最终演示路径

建议最终演示按下面路径走，能覆盖 4 个人的成果：

1. 管理员登录系统。
2. 管理端导入一批学生，查看导入日志。
3. 管理端新增一条政策，学生端搜索该政策。
4. 学生端查看党团流程和节点提醒。
5. 管理端按标签发布通知，学生端查看通知。
6. 学生端发起审批并上传附件。
7. 教师或管理员处理审批。
8. 领导端查看审批数量、待办、通知覆盖、学生画像等统计摘要。

## 13. 最小可交付版本

如果时间不够，优先保住下面这些：

| 模块 | 必须保留 | 可以延后 |
| --- | --- | --- |
| 登录权限 | 登录、角色、路由守卫 | 复杂数据范围 |
| 政策流程 | 政策检索、问答、流程提醒 | 真实全文解析 |
| 学生通知 | 学生导入、通知发布、画像展示 | Excel 解析、复杂标签规则 |
| 审批文件 | 审批创建、通过、驳回、附件记录 | 多级动态流程、正式对象存储 |
| 领导看板 | 基础统计卡片 | 复杂图表和导出报表 |

## 14. 为什么这个拆法比较均衡

- 1 号承担共享底座，虽然跨面广，但多数是基础能力，属于“先搭桥”的工作。
- 2 号负责 P0 核心体验，业务价值高，但页面和模型相对集中。
- 3 号负责管理端高频操作，涉及导入、维护、发布，工作量和复杂度都比较稳定。
- 4 号负责当前最空的模块，审批流和文件服务从零补齐，正好抵掉领导看板的展示和统计工作量。

这个组合的好处是：每个人都能独立推进，不会出现某个人一直等别人的接口。
